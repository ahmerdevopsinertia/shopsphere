What is Dependency Injection? (software engineering concept)

Read in detail from the ChatGPT chat as i cannot copy the entire content here

If an interviewer asked:

"Why use Dependency Injection?"

A strong answer would be:

"Dependency Injection reduces coupling by separating object creation from object usage. A class focuses only on its business responsibility and receives its dependencies from the framework or a container. This improves maintainability, testability, flexibility, and allows implementations to be replaced without changing the consuming class."

That answer demonstrates understanding beyond just using the framework.

Role	Example
Consumer	UsersRepository
Dependency	PrismaService
Container	NestJS Dependency Injection Container


---

How would we implement it in plain Node.js?

Like this we will implement

const prisma = new PrismaService();

const usersRepository = new UsersRepository(prisma);

const usersService = new UsersService(usersRepository);

const usersController = new UsersController(usersService);


---

How does NestJS implement it?


The Dependency Graph

This is a concept many developers don't know.

NestJS builds something called a Dependency Graph.

For your project today, it's approximately:

                 AppModule
                     │
         ┌───────────┴───────────┐
         ▼                       ▼
   PrismaModule            UsersModule
         │                       │
         ▼                       ▼
  PrismaService         UsersRepository
                               │
                               ▼
                         UsersService
                               │
                               ▼
                        UsersController

Why do enterprise systems rely on it?

	Loose coupling, can be replacable with any other source, can be testable


What are its trade-offs and alternatives?

Ignore it

Principle #3 — Build the Skeleton Before the Muscles

First create the complete structure of a feature (modules, folders, DTOs, services, controllers), then gradually add behavior one layer at a time.

===

This deserves another Team Rule

I'd like to formalize something we've already been practicing.

Team Rule #14 — Controllers Orchestrate, Services Decide
Controllers handle HTTP concerns (requests, DTO validation, responses).
Services contain business logic and make decisions.
Repositories only interact with the database.
Prisma only communicates with PostgreSQL.

Whenever you're unsure where code belongs, ask:

"Is this an HTTP concern, a business rule, or a persistence concern?"

That question alone will keep your architecture clean.

===


🏆 Team Rule #15

Let's add another engineering rule.

Trust, but Verify

Application Layer:

Validates business rules.
Gives meaningful error messages.

Database Layer:

Enforces data integrity.
Protects against race conditions and bugs.

Never assume application code alone is enough to protect your data.
